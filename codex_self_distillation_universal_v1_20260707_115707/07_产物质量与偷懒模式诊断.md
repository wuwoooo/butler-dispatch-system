# 产物质量与偷懒模式诊断

## 诊断对象

本轮重点诊断项目文档和未来 agent 输出，不评价业务功能好坏。

## 风险模式

| 风险 | 证据 | 影响 | 修复方式 |
| --- | --- | --- | --- |
| 文件数量冒充完成 | PDF 明确禁止；项目功能清单较长 | 人类难判断真实完成度 | 回执必须写验证命令、证据路径、未做事项 |
| 功能清单缺少验收证据 | `README.md` 列出大量功能，但没有每类功能的最新验证记录 | 后续接手成本高 | 增加 validator 和 smoke checklist |
| mock/dry-run 与真实完成混淆 | 项目含备份脚本、数据库脚本 | 可能误执行或误报完成 | 标记 `draft/mock/dry-run/real execution` |
| 高风险动作缺确认 | `scripts/backup-to-github.sh` | 可能推送远端或影响外部状态 | AGENTS 明确人工确认 |
| 业务规则散落 | `prisma/schema.prisma`、`lib/order-status.ts`、`lib/permissions.ts` | 新功能容易漏改状态/权限/日志 | 项目 skill 中集中规则 |
| 双端契约漂移 | `app/api/**`、`miniprogram/services/**` | 小程序与后台接口返回不一致 | 跨端修改必须检查请求层和 API 响应 |

## 工程类产物完成底线

任一代码变更完成前应至少说明：

- 修改了哪些文件。
- 是否涉及数据库 schema/migration/seed。
- 是否涉及 Web、小程序、API、权限、通知、日志、导出。
- 运行了哪些验证命令。
- 哪些验证没跑，原因是什么。
- 是否有回滚方式。

## 报告类产物完成底线

任一审计、总结、回执完成前应至少说明：

- 目标是什么。
- 使用了哪些证据路径。
- 哪些结论证据不足。
- 评分或优先级方法。
- 已生成 candidate 和未安装项。
- 下一轮可执行动作。
