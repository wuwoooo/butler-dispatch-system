# 自我蒸馏总报告

## 任务边界

本轮目标是把 PDF 中的自我蒸馏方法应用到当前项目，生成可审查、可回滚的候选产物。根据文档原则，本轮只创建 candidate，不覆盖正式 `AGENTS.md`、skill、自动化或业务代码。

## 项目判断

当前项目是旅游区酒店管家调配系统，包含 Next.js App Router 管理后台、微信小程序源代码、Prisma/MySQL 数据模型、统一 API 响应、JWT Cookie/Bearer 登录、角色权限矩阵、订单/多管家派单/请假/评价/财务/通知/异常记录等业务模块。

关键证据：

- `README.md`：项目功能、默认账号、API 列表。
- `package.json`：`lint`、`typecheck`、`miniprogram:build`、Prisma、脚本命令。
- `prisma/schema.prisma`：多角色、多管家派单、请假、评价、结算、通知、日志模型。
- `lib/response.ts`：统一 JSON 响应格式。
- `lib/auth.ts`：后台 Cookie 与小程序 Bearer 共用签名机制。
- `lib/permissions.ts`：角色权限矩阵。
- `lib/order-status.ts`：订单和管家派单状态流转。
- `miniprogram/services/request.ts`：小程序统一请求层。

## 主要发现

1. 项目没有正式 `AGENTS.md`，导致重复的业务约束、验证命令、双端联动规则缺少集中入口。
2. 项目已有脚本和分层，但缺少“完成前必须跑什么”的 agent 级验收规则。
3. README 功能覆盖广，但更像状态清单，缺少近期验收证据、风险、回滚、跨端影响说明。
4. 多管家订单、管家独立状态、请假冲突和小程序绑定属于高风险业务链路，适合沉淀专用审查 worker 和 validator。
5. 可发现 skill 多为通用或其它项目专用，当前项目没有“管家调配系统工程变更”专用 skill。

## 本轮生成的候选能力

- `09_AGENTS_md_patch_candidate.md`：项目级 agent 规则候选。
- `10_skill_patch_candidates/`：对已有 UI 验收 skill、PDF skill 的触发补丁建议。
- `11_new_skill_candidates/butler_dispatch_project_skill_candidate.md`：当前项目专用 skill 候选。
- `12_worker_subagent_candidates/`：通用 worker 和项目专用 worker 候选。
- `13_goal_longrun_candidates/butler_dispatch_quality_longrun_candidate.md`：长期质量巡检候选。
- `14_validators/`：Markdown 评分表、JSON schema、可运行脚本候选。
- `15_real_or_semireal_validation/`：一次半真实验证。

## P0 建议

1. 审查并安装项目 `AGENTS.md` 候选。
2. 保留 `14_validators/product_quality_validator_candidate.mjs`，作为后续报告、回执、handoff 的质量检查脚本。
3. 在任何触及订单状态、派单、请假、账号绑定、权限的任务中，强制给出验证命令和业务验收点。

## P1 建议

1. 新建“butler-dispatch-project”本地 skill，触发词覆盖“管家调配系统、派单、多管家、请假、小程序、后台管理、财务导出”等真实任务语言。
2. 为 Web 管理后台视觉验收接入截图矩阵，尤其是登录页、派单、订单详情、财务、通知。
3. 为小程序端引入独立构建和关键页面 smoke test 记录。

## 未做事项

- 未扫描用户所有历史 Codex 线程和外部账号记录，原因是本轮只基于当前项目和可访问本地证据，避免过度读取无关个人数据。
- 未安装正式 skill，原因是 PDF 明确要求先生成 candidate。
- 未运行数据库 migration 或真实备份脚本，原因是这类操作可能影响外部状态。
